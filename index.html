在任意地方部署 Web 应用。

从裸金属服务器到云虚拟机。

Kamal 提供零停机部署、滚动重启、静态资源桥接、远程构建、附属服务管理等能力，帮你用 Docker 在生产环境部署与管理 Web 应用。最初为 Rails 应用打造，但适用于任何可容器化的 Web 应用。

愿景

过去十多年，让 Web 应用部署更轻松的商业服务爆发式增长。Heroku 以卓越体验开了先河，并长期领先。如今我们有 Fly.io、Render 等优秀替代，托管 Kubernetes 也在 AWS、GCP、Digital Ocean 等地让事情更简单。但这些服务本质上是让你在云端以溢价租用算力。如果你想在自有硬件上运行，或只是希望将来能顺利迁移，就需要仔细权衡对商业平台的锁定程度——最好在账单把生意吞掉之前想清楚。

Kamal 希望把这些商业服务所开创的易用性，带到「在任意地方部署 Web 应用」。无论是 Digital Ocean、Hetzner、OVH 等没有托管溢价的低成本云，还是自建机房里的裸金属，对 Kamal 来说都一样。在配置文件里填入一组 IP，机器只需装好 Ubuntu 并添加 SSH 公钥，几分钟内就能跑起来。

这种方式带来极大的可移植性。你可以轻松把应用部署到多个云上，也可以用自有硬件打底，在季节性高峰前扩到云端再缩回来。工具链不绑定单一厂商时，选择空间会大得多。

归根结底，Kamal 要用不受商业平台绑定的开源工具，压缩上线生产的复杂度。不是压到零——如果基础 Linux 或 Docker 对你仍很吃力，托管服务可能更合适——但一旦熟悉这些概念，就可以用 Kamal 开干。

为什么不用 Capistrano、Kubernetes 或 Docker Swarm？

Kamal 本质上就是面向容器的 Capistrano，无需事先精心准备服务器。不必确保机器装了正确版本的 Ruby 或其他依赖——这些都在 Docker 镜像里。你可以启动一台全新的 Ubuntu（或其它发行版）服务器，加入 Kamal 的服务器列表，它会自动安装 Docker 并立刻运行。Docker 的分层缓存也让部署更快，服务器上动手更少。为 Kamal 构建的镜像还可用于 CI 或事后排查。

Kubernetes 体量庞大。在自有硬件上自建并不适合所有人。如果你想跑在别人的平台上——无论是 像 Render 那样透明托管，还是显式使用 AWS/GCP——它是不错的选择；但若你想在云与自有硬件之间自由切换，甚至混合使用，Kamal 简单得多。一切都一目了然，调用的就是基础 Docker 命令。

Docker Swarm 比 Kubernetes 简单，但仍建立在基于状态协调的声明式模型上。Kamal 有意采用类似 Capistrano 的命令式设计。

部署 Web 应用的方式有无数种，但这是我们在 37signals 用来把 HEY 以及其它曾托管在云上的应用迁回自有硬件 的工具箱——同时不放弃现代容器化带来的优势。

名称由来

Kamal 得名于古代阿拉伯航海工具：水手借北极星测定纬度以保持航向。（Kamal 曾用名 MRSK。）
